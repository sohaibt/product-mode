# Contributing to product-mode

Thanks for being here. `product-mode` is a living document, not a monument — contributions are welcome, especially the ones that disagree with me.

## What I'm Looking For

**High-value contributions:**

- **Proposed principles** — think there's an 8th? Open an issue with the principle, the failure mode it prevents, and a concrete test for "is it working."
- **Counter-examples** — cases where a principle is wrong, overkill, or misfires. These make the file sharper than agreement does.
- **Real-world case studies** — "we applied principle X on [feature], here's what changed." Short write-ups, linked or in-issue. These become the best documentation.
- **Edits for clarity** — shorter sentences, sharper tests, better examples. Brevity is a feature.
- **Translations** — if you want to port this to another language, open an issue first so we don't duplicate effort.

**Lower priority (but still welcome):**

- Typo fixes, formatting
- Tool-specific variants (Cursor, Lovable, Windsurf, etc.)
- Workshop or team-exercise formats built on the principles

## How to Contribute

**For discussion or a new idea:** open an issue. Describe what you'd change and why. Expect a conversation before a PR — `CLAUDE.md` is small, and every edit compounds.

**For small fixes (typos, broken links, formatting):** open a PR directly. Keep it to one change per PR.

**For principle-level changes:** open an issue first. Principles have a high bar — they need to prevent a real failure mode, be testable, and not overlap with an existing one.

## The Bar for a New Principle

Before proposing one, check:

- [ ] It names a failure mode not covered by the existing seven
- [ ] It has a concrete rule someone can follow, not just a vibe
- [ ] It has a test — "you know it's working when…"
- [ ] It survives the "would a senior PM or engineer roll their eyes?" check

If the answer to any of these is no, it's probably a sub-point under an existing principle — propose it there.

## Style Notes

- Prescriptive over prose. Checklists, tables, tests.
- Short sentences. Assume the reader is busy.
- No jargon unless it earns its place.
- No hedging words: *just, really, basically, actually, very.*

## Code of Conduct

Be useful. Be direct. Disagree with the work, not the person. That's it.
